# Case Study: VPN 内网路由问题 (2026-03-25~26)

## 数据摘要
- 81 个 session, 1878 分钟 (~31 小时), $8+
- 有效 session: 22 (27%)
- 无效/重复: 59 (73%)
- 问题经历 6 个阶段: 诊断 → 部署 → 内网路由 → 修隧道/503 → HK中转 → 客户端配置
- 最终状态: 仍在迭代

## 5 个根因 → 5 个进化规则

### Rule 1: 验证要一次做充分
**数据**: 12 个 session 是"试试/再试/刷新"的重复验证
**根因**: 每次只跑 1 个验证命令 (curl)，不充分
**规则**: 网络排查改动后，必须一次性跑完整验证套件:
```
ping <目标IP> -c 3
curl -sf <服务URL> --connect-timeout 5
traceroute <目标IP>
ssh <server> "wg show; iptables -L -n | head; journalctl -u <service> -n 5"
```
不要一条一条跑，不要等用户说"再试试"

### Rule 2: 方案切换必须记录 why
**数据**: 经历 5+ 个方案 (火山云隧道 → S2S 直连 → HK中转 → Clash → ...)，无记录
**根因**: 每个 session 独立，不知道上一个方案为什么不行
**规则**: 每次方案失败后，在 plan 文件中追加:
```
## 方案 N: [名称]
- 操作: ...
- 结果: 失败
- 失败现象: ...
- 根因: ...
- 排除理由: ...
```
下一个 session 先读这个文件，不重走老路

### Rule 3: 复杂排查先画拓扑
**数据**: 多组件联动 (Shadowrocket + WireGuard + AWS VPC + 火山云 + HK)
**根因**: 每次 session 重新理解拓扑，浪费时间
**规则**: 网络排查的第一个 session，先输出完整数据流拓扑:
```
客户端 → [协议/端口] → 中转 → [协议/端口] → 目标
```
标注每一跳的当前状态 (通/不通/未知)

### Rule 4: 先调研可行性再动手
**数据**: "能不能走 HK 转发" → 直接 SSH 改配置 → 发现不行 → 换方案
**根因**: 跳过了可行性调研，直接上手
**规则**: 新方案在动手前必须确认:
1. 技术上是否可行 (文档/先例)
2. 前提条件是否满足
3. 是否与现有配置冲突
4. 预计改动范围多大
只有 4 项都确认后才开始实施

### Rule 5: 人机交互步骤一次给完整清单
**数据**: agent 写命令 → 用户执行 → 报结果 → agent 再写 → ... (效率极低)
**根因**: 逐步给指令，每一步都是一个来回
**规则**: 需要用户手动操作的步骤，一次性给出完整操作清单:
```
请依次执行以下操作，完成后把所有输出一起发给我:
1. ssh root@server "command1"
2. 在 Shadowrocket 中切换到 xxx
3. curl http://target
4. 截图当前网络状态
```

## 对 Agent Workforce 的系统启示

1. **session ≠ task**: 这个问题是 1 个 task，但产生了 81 个 session。需要 task 聚合
2. **复杂排查类任务 agent 效率低**: 有效率只有 27%。应该标记为"需要人类主导"
3. **Done Spec 缺失导致无限循环**: 从未定义"什么算解决了"，所以一直在试
