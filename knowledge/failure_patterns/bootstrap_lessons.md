# Bootstrap Lessons — 从平台创建过程中学到的

来源: 2026-03-25 Agent Workforce 创建对话

## 用户决策模式

### 1. 先调研再动手
用户不会直接要求"帮我建一个系统"。他的模式是:
- 先给背景信息 (ChatGPT 对话截图)
- 再给参考项目 (rd-agent)
- 要求调研替代方案
- 结合本地项目现状分析
- 确认方向后才说"动手"

**教训**: 不要跳过调研直接给方案。用户需要看到决策依据。

### 2. 约束驱动设计
用户主动提出的硬约束:
- "所有部署相关的事情都需要人类确认"
- "现阶段我们先用 Claude 来做，不引入外部的 API"
- "每个 Agent 的新增需要人类确认"
- "规则的修改需要人类二次确认"
- "我们用一个飞书群来进行对人类的反馈交互"

**教训**: 用户对安全边界非常敏感。任何涉及生产环境、新增实体、规则变更的操作必须有人类确认。

### 3. 反馈必须低摩擦
用户反复强调的核心需求:
- 80% 的场景什么都不做也要能跑
- 反馈最多 3 秒（按一个数字键）
- 不要强制每次都填表

**教训**: 任何需要用户额外操作的功能，摩擦越低越好。隐式信号 > 快捷评分 > 结构化表单。

### 4. 执行隔离 + 数据聚合
用户明确要求:
- 不同项目在执行层隔离（不能跨目录操作）
- 但在数据层聚合（所有 trace 统一存储，跨项目学习）

**教训**: 这是安全和智能的平衡。隔离防止出错，聚合促进进化。

## 架构决策记录

| 决策 | 选择 | 否决的替代方案 | 原因 |
|------|------|--------------|------|
| 框架选型 | 不引入新框架，在 Claude Code 上扩建 | rd-agent / LangGraph / CrewAI | 一个人维护不了额外框架 |
| 夜间评测 | 本地 Qwen3.5-35B | 云端 Claude/GPT | 零成本 + 隐私 |
| 反馈渠道 | 飞书群 | Telegram / 终端 | 用户已有飞书生态 |
| Agent 数量 | 5+1+1 (不超过 Google 研究的 4 agent 阈值 + orchestrator) | 更多细分 | 超过 4 个性能退化 |
| 模型分配 | Opus(调度) + Sonnet(执行) + Haiku(infra) | 全 Sonnet | 成本降 40-60% |
| Trace 存储 | JSONL 按日分片 | SQLite / MongoDB | 最简单，够用 |
| Profile 格式 | YAML | Python class / JSON | 人类可读可编辑 |

## 用户对 Agent 行为的期望

从对话中提取的隐式期望:
1. **不要过度工程** — "不加以后可能用到的代码"
2. **先读后改** — 不读文件就改 = 盲改
3. **最小变更** — 不顺手重构
4. **契约先行** — 跨端任务先定义 API contract
5. **上游只读** — 绝不修改原始数据
6. **部署分离** — agent 写脚本，人类执行

## 本对话的进化价值

这个对话可以作为:
1. **Orchestrator 的 golden example** — 如何理解用户目标、拆分任务、做技术选型
2. **系统设计类任务的参考** — 调研 → 对比 → 决策 → 实现的完整流程
3. **用户偏好的基准** — 用户如何思考、什么打动他、什么让他不满
